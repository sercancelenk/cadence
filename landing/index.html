Why Cadence is different

Four trade-offs that productivity tools usually skip. We pick the unfashionable side on each of them.

100% local-first, syncs without a cloud

Every keystroke is written to your machine, not to a server. Atomic fsync'ed writes, 50 rolling auto-snapshots, refuse-to-overwrite guard, enriched in-app recovery UI (per-snapshot counts, Reveal in Finder, restore confirmation). If a boot loads meaningfully less than your last save, an amber banner tells you before you panic. When you want two devices to share a workspace, an opt-in LAN server lets them talk directly — your bytes stay between your phone and your laptop.

Encrypted on disk, encrypted on the wire

The desktop data file is wrapped in AES-256-GCM with a key derived from your account password via scrypt. Notes get an additional opt-in passphrase lockbox. LAN sync rides over TLS 1.2+ with a self-signed cert and a 192-bit bearer token — bytes are encrypted whether they're sitting still or in motion.

AI when you want it

Bring your own API key for Anthropic Claude, OpenAI or Google Gemini. Calls go directly from your device to the provider — no proxy, no Cadence-side telemetry. Turn AI off completely and nothing else changes.

Open source, no upsell

The codebase is on GitHub. No accounts in the cloud, no premium tier, no waitlist. If you don't like a behaviour you can fork it. The desktop app auto-updates straight from GitHub Releases.

Bidirectional auto-sync

Paired devices stay in step in the background — push first, pull after — on launch, on window focus, on visibility change, when the network comes back. SHA-256 ETags decide direction so we never overwrite a remote that moved on, and never re-download bytes you already have.

Optimistic concurrency

The host rejects pushes with a 412 Precondition Failed when an ETag is stale; the app surfaces an explicit "Pull host's version first?" prompt. Last-write-wins becomes a deliberate decision, not an accident. Background auto-sync backs off silently so manual resolution stays the single source of truth.

Cloud sync · End-to-end encrypted · Your Drive, your keys

When you can't be on the same Wi-Fi.

Google Drive sync gives you the convenience of cloud backup without the privacy trade. Cadence wraps every snapshot with AES-256-GCM (key derived via PBKDF2-SHA-256, 200k iterations) before it leaves your device. Google stores opaque ciphertext in a hidden folder visible only to Cadence. We don't run a server. We don't see a byte.

Encrypted before upload

Your sync passphrase derives a 256-bit AES key locally via PBKDF2-SHA-256 (200,000 iterations). The snapshot is encrypted with AES-256-GCM, optionally gzip- compressed, and uploaded as a versioned binary blob. Drive stores ciphertext. Without your passphrase, nobody can decrypt it — not Google, not us, not a device that gains access to your Drive.

AES-256-GCM · PBKDF2 200k · gzip

Hidden appdata folder

Cadence uses Drive's appDataFolder scope — a per-app hidden folder invisible in your Drive UI, inaccessible to other apps. No clutter, no accidental share, no quota surprise (counts against your own Drive). Uninstalling Cadence doesn't lose the backup.

drive.appdata scope only · OAuth PKCE

Background auto-sync, smart conflict UX

On launch, on focus, when the network comes back — Cadence quietly checks Drive's version field and reconciles. A real conflict surfaces an inline Pull first / Override remote resolver so you decide; auto-sync never makes a destructive choice for you. Built-in retry + exponential backoff rides through transient 5xx / rate-limit responses.

Silent token refresh

Once signed in, Cadence handles Google's token expiry transparently — your access token gets rotated in the background using the refresh token, with the user's email displayed so you always know which Drive account holds the backup. Sign out revokes server-side and clears local state.

Five minutes, in-app, no rebuild.

Cadence does not run a hosted Google project for you, so you connect your own Google Cloud project once — it stays under your account and keeps your data isolated from every other Cadence user. After that first connect, it's transparent.

  1. 1
    Create a free Google Cloud project and enable the Drive API. Cadence asks for the drive.appdata scope only — a hidden per-app folder, not your real Drive.
  2. 2
    Create an OAuth Web client ID in that project. Copy the value that ends with .apps.googleusercontent.com.
  3. 3
    Open Cadence → Settings → Cloud sync, paste the client ID, click Save. The card flips to "Sign in with Google" — sign in with the Google account that holds the project.
  4. 4
    Pick a sync passphrase (8+ chars). This is what derives the AES key. Cadence never sees it, Google never sees it. Write it down — it's the only way to decrypt your backup, which is exactly the point of end-to-end encryption.
  5. 5
    Repeat steps 3–4 on every other device you want to sync (mobile PWA, second laptop). Same client ID, same Google account, same passphrase. Background auto-sync handles the rest.

Everything happens inside the app. No .env, no rebuild, no command-line — the client ID is stored in localStorage and used at runtime. If you ever want to revoke it, expand the same card and click Clear saved ID.

What "local-first" actually means here

Some "local-first" apps mean "we sync your data through our cloud and call it caching". Cadence is the stricter version.

Where your data lives

On macOS: ~/Library/Application Support/Cadence/. On Windows: %APPDATA%\Cadence\. On Linux: ~/.config/cadence/. Trashing the app does not delete your data — the data folder lives outside the application bundle.

What survives a crash

Every save opens a sibling .tmp, writes, calls fsync, then atomically renames. A power loss mid-write leaves either the old file or the new file — never a half-written one. Worst case: ≤ 400 ms of unflushed typing.

Backups, by default

Every save snapshots the previous file into backups/<userId>/. 50 rolling slots labelled launch / post-login / pre-save / pre-pwchange / pre-restore — oldest pruned automatically so disk use stays bounded. Settings → Backup & recovery combines manual export/import with snapshot restore; each snapshot shows teams / lists / tasks / notes counts and restores with a clear "Loaded: …" confirmation.

Archive without deleting

Notes and to-dos support an Active · Archived segment. Shelving hides items from default views (and archived notes from ⌘K search) while keeping them in your encrypted workspace until you delete them. Settings shows a per-area storage breakdown so you can see what archived content still occupies.

Efficient desktop storage

Large workspaces are split into monthly files on disk for performance, while the base file keeps a full copy for safety if you ever open an older Cadence build. You never need to manage these files — the app reads and writes them automatically.

Integrity check on boot

After each successful save, Cadence remembers how much content you had. If the next launch loads a meaningfully smaller workspace, an amber banner spells out the before/after counts and links straight to recovery — catching "my data vanished" before it becomes a support ticket.

Refuse-to-overwrite

If the data file exists but can't be decrypted with the current session key, the writer refuses. You can't accidentally destroy a locked workspace with one character's worth of state after a bad login.

Optional LAN sync (off by default)

Settings → Multi-device sync exposes an HTTPS server (TLS 1.2+, self-signed cert generated on the host) protected by a 192-bit bearer token and an ETag conflict guard. QR-code pairing on phones, bidirectional auto-sync once paired. See the Sync section above for the full story.

No telemetry. Period.

Cadence makes zero network requests on its own. The auto-updater hits GitHub Releases. The AI assistant hits whichever provider you configured. Nothing else dials home.
